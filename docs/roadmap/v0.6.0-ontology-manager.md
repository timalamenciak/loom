# v0.6.0 — Ontology Manager

**Epic:** E2: Ontology · **Depends on:** v0.5.0 · **Unlocks:** v1.0.0 path

## Goal

Admins can load LinkML YAML files (NAOs — Native Annotation Ontologies) as browsable term vocabularies alongside OBO/OWL sources. A browser UI replaces the `load_ontology` management command for day-to-day operations. Slot → ontology mapping is configurable through the form builder rather than `loom_ui.yaml` hand-editing. A lightweight update check detects when upstream ontology files have changed.

## Baseline

- `apps/ontology/loaders.py` — `load_ontology()` handles OBO/OWL via pronto. No LinkML YAML path yet.
- `apps/ontology/models.py` — `OntologyRelease`, `OntologySnapshot`, `OntologyTerm`, `AdHocOntologySource` all exist.
- `config/ontologies.yaml` — 8 ontologies configured (ENVO, ELMO, PATO, GO, CHEBI, BFO, RO, ECO).
- `apps/ontology/services.py` — trigram + Wikidata search fully implemented.
- No ontology admin views exist; no LinkML loader; no update checking.

## Implementation Steps

### Step 1 — LinkML/NAO loader

**File:** `apps/ontology/loaders.py`

Extend `load_ontology()` to detect LinkML YAML input (`.yaml` extension that is not an OBO file — detect by checking for `id:` at top level with a schema URL, or by trying pronto parse and falling back). When detected:

1. Parse with `linkml_runtime.utils.schemaview.SchemaView(yaml_str)`
2. Iterate `schema_view.all_enums().values()` — each `EnumDefinition` is a vocabulary
3. For each `EnumDefinition`, iterate `enum_def.permissible_values.items()`
4. Create an `OntologyTerm` per `PermissibleValue`:
   - `curie` = `f"{enum_def.name}:{pv_text}"` (slugified)
   - `label` = `pv.text`
   - `synonyms` = `[]`
   - `definition` = `pv.description or ""`
5. Bulk insert via `OntologyTerm.objects.bulk_create(ignore_conflicts=True)`
6. Support an optional `root_terms` filter (list of enum names to include; if empty, load all enums)

### Step 4 — Ontology browser

`GET /admin/ontologies/<pk>/browse/` renders a search box. HTMX GET to `/admin/ontologies/<pk>/search/?q=<query>` returns a partial of up to 20 `OntologyTerm` rows (curie, label, definition snippet). Uses the existing `apps/ontology/services.search_terms()` function.

This is a read-only diagnostic tool — useful for verifying a load succeeded and finding specific terms.

### Step 5 — Slot → ontology mapping in form builder

Extend the form builder (Step 3 of v0.5.0) to include ontology routing in the per-slot config panel. Each slot's config panel gains an "Ontology sources" multi-select populated from `OntologyRelease.objects.filter(status='ready')` (name + prefix). Selecting ontologies adds them to the `ontology_routing` JSONField in `SchemaUIConfig` in the simplified format `{slot_name: [{prefix: "ENVO"}, {prefix: "ELMO"}]}`.

`LoomSchemaView.form_spec()` already handles `ontology_routing`; extend the format parsing to also handle this flat list format alongside the existing conditional format.

### Step 6 — Ontology update checking

**File:** `apps/ontology/models.py`, new management command

Add two fields to `OntologyRelease`:
- `upstream_url` (URLField, nullable) — the canonical fetch URL
- `source_etag` (CharField, max_length=255, blank=True) — last-seen ETag or Last-Modified header value
- `update_available` (BooleanField, default=False)

Management command `check_ontology_updates`:
- Iterate `OntologyRelease.objects.filter(upstream_url__isnull=False)`
- Send `HEAD` request to `upstream_url` (urllib, 10s timeout)
- Compare `ETag` or `Last-Modified` header to `source_etag`
- If different: set `update_available=True`, print notice
- Fails soft on network error (logs warning, continues to next)
- Exit 0 if all up to date, exit 1 if any updates found
