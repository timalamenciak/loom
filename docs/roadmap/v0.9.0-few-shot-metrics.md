# v0.9.0 — Few-Shot & Metrics

**Epic:** E4: LLM · **Depends on:** v0.8.0 (LLM Pipeline Core)

## Goal

Few-shot examples drawn from real approved annotations improve proposal quality — the prompt shows the model what a good annotation looks like for this specific project's content. Acceptance-rate and edit-distance metrics give admins a defensible answer to "is the LLM saving time?".

## Baseline

- `ClaudeProposer.propose()` works but uses zero-shot prompting (system prompt + document text only).
- `ProposerConfig` exists with `max_proposals_per_document`.
- `Edge` and `Node` models exist with `status` and `origin` fields.
- No `FewShotExample` model. No metrics model. No metrics UI.

## Implementation Steps

### Step 2 — Few-shot serializer

**File:** `apps/llm/prompt_builder.py`

`serialize_example(edge) -> dict` — serialize a `FewShotExample`'s edge to the same JSON format the LLM is expected to output:

```python
{
    "subject": {"name": edge.subject.name, **edge.subject.data},
    "object": {"name": edge.object.name, **edge.object.data},
    "edge": {**edge.data},
    "source_text": edge.data.get("original_sentence", "")
}
```

`build_few_shot_messages(examples: list[FewShotExample]) -> list[dict]` — produce alternating human/assistant message pairs for inclusion before the main user prompt:

```python
[
    {"role": "user", "content": f"Extract claims from: {example.edge.data.get('original_sentence', '')}"},
    {"role": "assistant", "content": json.dumps(serialize_example(example.edge))},
    ...
]
```


### Step 3 — Inject few-shot into `ClaudeProposer`

**File:** `apps/llm/claude_proposer.py`

In `propose()`, before calling the API:
1. Fetch `FewShotExample.objects.filter(project=document.project).order_by('display_order').select_related('edge__subject', 'edge__object')[:config.max_shots]`
2. Build few-shot messages via `build_few_shot_messages(examples)`
3. Prepend to the messages list before the main user message

```python
messages = few_shot_messages + [{"role": "user", "content": doc_text}]
```

### Step 4 — Few-shot selection UI

**File:** `apps/llm/views.py`, template `templates/llm/few_shot_selector.html`

`FewShotSelectorView` at `GET/POST /admin/projects/<pk>/llm-config/few-shot/`:
- GET: list all `Edge.objects.filter(graph__document__project=project, status__in=['complete', 'gold']).select_related('subject', 'object', 'graph__document')[:200]`. Show each with a checkbox. Already-selected examples (from `FewShotExample`) are pre-checked. Show a short "source_text" preview per edge.
- POST: for each checked edge, `FewShotExample.objects.get_or_create(project=project, edge=edge, defaults={'selected_by': request.user, 'label': request.POST.get(f'label_{edge.pk}', '')})`. For unchecked, `FewShotExample.objects.filter(project=project, edge=edge).delete()`. Redirect back with a success message showing count.

Admin can set a `label` for each selected example to remind themselves why it was chosen.

### Step 5 — `ProposalOutcome` tracking

**File:** `apps/llm/models.py`

Track what happened to each LLM proposal so you can compute acceptance rates:

Fields:
- `edge` — OneToOneField to `Edge` (CASCADE)
- `accepted_at` — DateTimeField, nullable
- `rejected_at` — DateTimeField, nullable
- `edit_distance` — IntegerField, nullable (number of JSONB fields changed between proposal and accepted state)
- `time_to_review_seconds` — IntegerField, nullable (seconds from Edge.created_at to outcome)

The accept/reject views (from v0.8.0) should update `ProposalOutcome` on each action.

`edit_distance` computation in the accept view: diff the original edge JSONB snapshot against the edge JSONB at accept time. Count the number of top-level keys whose value changed. A diff of 0 means the annotator accepted without changes.

### Step 6 — Metrics dashboard

`GET /admin/projects/<pk>/llm-metrics/` — a read-only stats page for project admins:

- Total proposals generated (count of `Edge.objects.filter(origin='llm_proposed')`)
- Acceptance rate (accepted / total) as a percentage with a colour-coded bar
- Average edit distance for accepted proposals
- Average time-to-review in minutes
- Proposals per document (accepted vs. rejected breakdown)

No chart library needed — render as a simple table + HTML progress bars (CSS `width` attribute).
