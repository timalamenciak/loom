# v0.7.0 — Module Update Manager

**Epic:** E3: Updates · **Depends on:** v0.4.0 · **Can run in parallel with:** v0.5.0, v0.6.0

## Goal

Loom automatically detects new schema and ontology releases in the background, persists the check results, and shows a notification banner to admin users. Admins can review a diff of what changed and apply the update in one click without CLI access.

## Baseline

- `apps/schemas/upstream.py` — GitHub Releases API client exists; polls causalmosaic CAMO repo for latest release, computes slot/enum diffs.
- `apps/schemas/management/commands/check_schema_updates.py` — polls GitHub, prints diff, exits 0/1/2. Works but is manual-only.
- `apps/schemas/management/commands/update_schema.py` — downloads and loads a specific release. Works but is manual-only.
- `apps/ontology/models.py` — `update_available` field exists (from v0.6.0 step 6).
- No `UpdateCheckRecord` model. No admin notification. No update UI.

## Implementation Steps

### Step 1 — `UpdateCheckRecord` model

**File:** `apps/schemas/models.py` (or a new `apps/updates/models.py` if you prefer a dedicated app)

Fields:
- `module_type` — CharField with choices `schema` / `ontology`
- `module_name` — CharField (e.g. `"camo"` or `"envo"`)
- `current_version` — CharField
- `available_version` — CharField (empty if no update)
- `is_update_available` — BooleanField, default False
- `checked_at` — DateTimeField, auto_now
- `release_notes` — TextField (raw JSON string from GitHub API, or summary)
- `diff_summary` — JSONField (structured diff: `{added_slots, removed_slots, changed_enums}` for schema; `{etag_changed: true}` for ontology)
- `download_url` — URLField (nullable; the asset URL to fetch on apply)

One record per `module_name`. Upserted (not appended) on each check.

### Step 2 — `check_all_updates()` service

**File:** `apps/schemas/update_service.py` (new file)

A single function that wraps all update checks and persists results to `UpdateCheckRecord`:

1. **Schema check**: call existing `apps/schemas/upstream.py` check logic. Extract `current_version`, `available_version`, `is_available`, `diff_summary` (added/removed/changed slots), `download_url` (GitHub asset URL). Upsert `UpdateCheckRecord(module_type='schema', module_name='camo', ...)`.

2. **Ontology check**: iterate `OntologyRelease.objects.filter(upstream_url__isnull=False)`. For each, send a `HEAD` request (urllib, 5s timeout). If ETag changed, upsert `UpdateCheckRecord(module_type='ontology', module_name=release.name, is_update_available=True, diff_summary={'etag_changed': True})`.

Fails soft throughout: network errors log a warning and continue; `UpdateCheckRecord` is still written with the last-known state.

Create a management command `check_all_updates` that calls `check_all_updates()` and prints a summary.


### Step 3 — Background check on admin dashboard load

**File:** `apps/schemas/views.py` or middleware

On admin dashboard page load: check `django.core.cache.cache.get('loom:update_check_last_run')`. If absent (first load or post-12-hour expiry): spawn a daemon thread (`threading.Thread(target=check_all_updates, daemon=True).start()`) and immediately set `cache.set('loom:update_check_last_run', now(), 43200)`. This gives auto-check behavior without requiring Celery.

The check runs in the background — it won't delay page load. Results are available for the next page load via `UpdateCheckRecord`.

```
# Prompt for Claude Code:

for staff users), add the following logic at the start of the GET handler:
from django.core.cache import cache

    cache.set('loom:update_check_last_run', True, 43200)  # 12 hours

    t.start()
Do not await the thread; do not block the response on it. Write a test
test_admin_dashboard_triggers_check_when_stale (cache empty → thread started,
confirmed via mock on check_all_updates) and test_admin_dashboard_no_check_when_fresh
(cache populated → check_all_updates NOT called).
```

### Step 4 — Admin notification banner

**File:** `templates/base_admin.html` (or the base template used by admin pages)

At the top of the page (below nav, for `is_staff` users only):

```html

      {% for u in updates %}

        · <a href="{% url 'schemas:apply_update' u.pk %}">Update now</a>

    {% endif %}
  {% endwith %}

```

Pass `pending_updates = UpdateCheckRecord.objects.filter(is_update_available=True)` via a template context processor so it's available on every admin page without modifying each view.

### Step 5 — Diff viewer

`GET /admin/updates/<pk>/diff/` renders `templates/schemas/update_diff.html`. Shows:
- Module name, current version, available version
- For schema diffs: three-column table of added slots (green), removed slots (red), changed enums (amber)
- For ontology diffs: term count change (if available) and ETag change notice
- "Apply Update" form (POST button) and "Dismiss" link

### Step 6 — Apply update workflow

`GET /admin/updates/<pk>/apply/` — confirmation page showing the diff + a checkbox "Activate new schema immediately" + a "Confirm Update" button.

`POST /admin/updates/<pk>/apply/` (with `confirm=1`):
- For `module_type='schema'`: download bytes from `UpdateCheckRecord.download_url` (urllib, 30s timeout), call the existing `load_schema` flow to create a `SchemaVersion`, optionally activate
- For `module_type='ontology'`: call `load_ontology(release.name, force=True)` for the matching `OntologyRelease`
- On success: `UpdateCheckRecord.is_update_available = False`, redirect to success page
- Guard against double-submit: check if the version is already loaded before downloading

## Acceptance Criteria

- [ ] Admin dashboard shows a banner within 12 hours of a new CAMO schema release on GitHub
- [ ] Admin can view a slot-level diff before applying update
- [ ] Applying an update downloads and loads the new schema without CLI access
- [ ] Updating schema does not affect existing `CausalGraph` rows (they remain pinned to their `schema_version`)
- [ ] Annotator-facing pages never show the update banner
