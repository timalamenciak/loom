# v0.5.0 — Visual Form Builder

**Epic:** E1: Schema · **Depends on:** v0.4.0 (SchemaUIConfig model) · **Unlocks:** v0.6.0 (Ontology Manager)

## Goal

Admins can configure the annotation form layout for any uploaded schema entirely through the browser — grouping slots into layers, marking slots as hidden, setting widget types, adding help text, and configuring ontology routing — without editing any YAML file. Non-technical admins setting up Loom with a new LinkML schema should be able to produce a working annotation form from scratch.

## Baseline

- `SchemaUIConfig` model exists (from v0.4.0) with `layers`, `widget_overrides`, `globally_hidden_slots`, `slot_help_text`, `ontology_routing` JSONFields.
- `LoomSchemaView.form_spec(project=project)` consumes `SchemaUIConfig` (from v0.4.0).
- `templates/schemas/schema_demo.html` renders a form preview given a class name — reuse this for the form builder preview panel.
- Alpine.js and HTMX are in `static/` — no new JS dependencies.

## Implementation Steps

### Step 1 — Form builder scaffold view

**File:** `apps/schemas/views.py`, `apps/schemas/urls.py`

`FormBuilderView` at `GET /admin/schemas/<pk>/form-builder/`:
1. Load `SchemaUIConfig.for_schema_version(sv)` — the starting configuration
2. Call `LoomSchemaView(sv).all_class_slots(class_name)` for each annotatable class (CausalEdge, CausalNode, or whatever the active schema defines) to get all slot names and their metadata
3. Compute "unassigned slots" = all schema slots minus slots already present in `config.layers`
4. Render `templates/schemas/form_builder.html` with: `sv`, `config`, `class_slots`, `unassigned_slots`

The template uses Alpine.js for all in-page state (layers list, slot assignments, slot configs). HTMX handles the save POST.

```

GET at /admin/schemas/<pk>/form-builder/: load SchemaVersion by pk, call
SchemaUIConfig.for_schema_version(sv) to get config, call
linkml_runtime.SchemaView on sv.yaml_content to enumerate all classes and their
```

### Step 2 — Alpine.js form builder template

**File:** `templates/schemas/form_builder.html`

Two-panel layout:
- **Left panel** — "Layer configuration": list of layers (from `config.layers`), each showing its slots in order. Slots can be dragged between layers using Alpine.js `$el.addEventListener('dragstart'/'drop')` (native HTML5 drag-and-drop, no library). A "+ Add Layer" button appends a new layer with a default name. Layer names are editable inline.
- **Right panel** — "Available slots": slots not yet in any layer. Clicking a slot opens a config panel (see Step 3).
- **Sticky footer** — "Save" button sends Alpine state as JSON to `/admin/schemas/<pk>/form-builder/save/` via HTMX POST.
- **Preview** button — HTMX GET to `/admin/schemas/<pk>/form-builder/preview/` swaps in a form preview panel using `schema_demo.html` partial.

All state lives in `x-data` on the root div. No page reload on reorder or slot config.

```
# Prompt for Claude Code:
Create templates/schemas/form_builder.html extending base.html. Use Alpine.js
x-data to hold: layers (array of {name, slots: [{name, hidden, widget, help_text,
required_override}]}), unassigned (array of slot names). Initialize both from
JSON embedded in a <script type="application/json"> tag populated from the view's
config_json. Implement drag-and-drop between layers and the unassigned panel using
native HTML5 draggable="true" and ondrop events managed through Alpine.js
$dispatch and $watch. Layer names must be editable via <input x-model="layer.name">.
A "+ Add Layer" button does layers.push({name: 'New Section', slots: []}). A
"Remove Layer" button moves that layer's slots back to unassigned. The "Save" button
serializes the Alpine state to JSON and POSTs to /admin/schemas/<pk>/form-builder/save/
via fetch (not HTMX, since we need to serialize the Alpine state). Do not add
Sortable.js or any external drag-drop library.
```

### Step 3 — Per-slot configuration panel

Within the form builder template, clicking any slot name opens an Alpine `x-show` panel (slide-in from right, or inline expand). The panel contains:
- **Hidden toggle** — checkbox; if checked, slot is added to `globally_hidden_slots` on save
- **Widget override** — `<select>` populated from `widget_choices` passed by the view; default shows the schema-inferred widget
- **Help text** — `<textarea>` for the placeholder string
- **Required override** — checkbox to force-required even if schema marks optional

All state changes update the corresponding slot object in the Alpine `layers` array immediately (reactive). The panel closes on click-outside or an Escape key listener.

### Step 4 — Save endpoint

`POST /admin/schemas/<pk>/form-builder/save/` receives JSON body:
```json
{
  "layers": [{"name": "...", "slots": [{"name": "...", "hidden": false, "widget": null, "help_text": ""}]}],
  "ontology_routing": {}
}
```

Deserializes and calls `SchemaUIConfig.objects.update_or_create(schema_version=sv, project=None, defaults={...})`. Returns JSON `{"ok": true}` or `{"error": "..."}`.

```
# Prompt for Claude Code:
Create FormBuilderSaveView in apps/schemas/views.py at POST
/admin/schemas/<pk>/form-builder/save/ (csrf_exempt is NOT needed — include
CSRF token in the fetch call's headers as X-CSRFToken from the cookie).
Parse request body as JSON. Extract layers list. Build SchemaUIConfig fields:
- layers: [{name, slots: [name, ...]}] (strip per-slot config from slot objects)
- globally_hidden_slots: collect all {hidden: true} slot names across all layers
- widget_overrides: {slot_name: widget} for any slot with non-null widget
- slot_help_text: {slot_name: help_text} for any slot with non-empty help_text
Call SchemaUIConfig.objects.update_or_create(schema_version=sv, project=None,
defaults=computed_fields). Return JsonResponse({'ok': True}) on success,
JsonResponse({'error': str(e)}, status=400) on exception.
Write test: test_save_creates_config (POST valid JSON → SchemaUIConfig row with
correct layers), test_save_updates_existing (second POST → updates, no duplicate row),
test_save_requires_staff (anonymous → 403).
```

### Step 5 — Live form preview

`GET /admin/schemas/<pk>/form-builder/preview/?class=CausalEdge` renders the annotation form for `class_name` using the *in-progress* `SchemaUIConfig` (loaded from DB, which may not have been saved yet from the in-progress Alpine state). Returns an HTML partial (no base template wrapper) for HTMX swap. Reuse the existing `schema_demo.html` partial.

Since the preview renders the saved state (not the in-flight Alpine state), the "Preview" button should first auto-save, then fetch the preview. The save endpoint returns `{"ok": true}` and the JS then issues a second fetch for the preview.

### Step 6 — Export/import form config as YAML

`GET /admin/schemas/<pk>/form-builder/export/` — serialize current `SchemaUIConfig` to YAML (same format as `loom_ui.yaml`) and return as a file download.

`POST /admin/schemas/<pk>/form-builder/import/` — accept a YAML file upload, parse it, and populate `SchemaUIConfig` via `update_or_create`. Validate that all slot names referenced exist in the schema before saving.

This gives teams a way to share form configurations and gives power users an escape hatch for advanced conditional ontology routing that the builder UI doesn't expose.

```
# Prompt for Claude Code:
Create FormBuilderExportView and FormBuilderImportView in apps/schemas/views.py.
Export: serialize SchemaUIConfig.for_schema_version(sv) to a dict matching the
loom_ui.yaml structure (layers, ontology_routing, widget_overrides,
globally_hidden_slots, slot_help_text), dump with yaml.safe_dump(), return as
HttpResponse with Content-Type: application/x-yaml and
Content-Disposition: attachment; filename=form_config_{version}.yaml.
Import: accept multipart POST with a 'config_file' field, parse with yaml.safe_load(),
validate that all slot names in layers exist in the active schema (use
LoomSchemaView(sv).all_slots()), call SchemaUIConfig.objects.update_or_create with
parsed values. On validation error, return 400 with error list.
Write tests: test_export_yaml_parseable (GET → valid YAML), test_import_valid_yaml
(POST valid YAML → SchemaUIConfig updated), test_import_unknown_slots (POST YAML with
slot that doesn't exist in schema → 400 with error listing the unknown slot name).
```

## Acceptance Criteria

- [ ] Admin can configure layers, slot order, and slot visibility for a brand-new (non-CAMO) schema without editing any file
- [ ] Form preview shows the exact widget layout annotators will see
- [ ] Exporting the CAMO form config produces a YAML semantically equivalent to the current `loom_ui.yaml`
- [ ] Importing that YAML back produces an identical form_spec output
- [ ] Existing CAMO annotation workflow is unaffected (loom_ui.yaml still loads as default)
