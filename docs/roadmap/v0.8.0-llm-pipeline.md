# v0.8.0 — LLM Pipeline Core

**Epic:** E4: LLM · **Depends on:** v0.5.0 (stable form_spec) · **Unlocks:** v0.9.0 (Few-Shot)

## Goal

A working `ClaudeProposer` generates draft annotations from document text using the Claude API. The proposer is schema-agnostic — its prompt is derived entirely from `LoomSchemaView.form_spec()`, so it works for any uploaded LinkML schema. Per-project configuration controls whether proposals are generated and when. Annotators see a review queue and can accept or reject each proposal.

## Baseline

- `apps/llm/proposer.py` — `Proposer` protocol + `NoOpProposer` + `ProposedNode`/`ProposedEdge` dataclasses. Disabled by default (`LLM_PROPOSALS_ENABLED = False`).
- `apps/annotation/services.py` — `create_node()`, `create_edge()` already accept `origin` and `status` parameters.
- `apps/annotation/models.py` — `Edge` has `origin` and `status` fields.
- `apps/documents/models.py` — `Document` has `canonical_text` (extracted text with character offsets).
- No `ProposerConfig` model. No real proposer. No review UI.

**New dependency:** `anthropic` (Anthropic Python SDK). Add to `pyproject.toml`.

### Step 2 — Schema-agnostic prompt builder

**File:** `apps/llm/prompt_builder.py` (new file)

`build_system_prompt(form_spec: dict) -> str` — given the output of `LoomSchemaView.form_spec()`, produce a system prompt that describes every non-hidden slot:

- For each class (node, edge): a section listing slot name, description, range/type, cardinality, and for enum slots the list of allowed values with their descriptions
- For ontology slots: "Use a CURIE from prefixes: ENVO, ELMO, ..." (extracted from slot spec's ontology routing)
- For required slots: mark clearly
- End with: explicit output format instruction (JSON array of `{subject, object, edge, source_text}` objects)

Keep the prompt under ~3000 tokens for small schemas to leave room for document text and few-shot examples.

```

where form_spec is the dict returned by LoomSchemaView.form_spec(). The function
should: iterate form_spec['layers'] (each layer has 'name' and 'fields'), for each
field (slot) build a description line: "- slot_name (required|optional, type): description.
[Allowed values: val1, val2, ...] [Use ontology prefix: PREFIX1, PREFIX2]". Return
a complete system prompt string that: (1) introduces the task ("You extract causal
claims from scientific text"), (2) lists all node slots and edge slots with their
descriptions and constraints, (3) specifies output format as a JSON array of objects
with keys 'subject' (node data dict), 'object' (node data dict), 'edge' (edge data dict),
'source_text' (quoted sentence containing the claim). Do not hardcode any CAMO slot

### Step 3 — `ClaudeProposer` implementation

**File:** `apps/llm/claude_proposer.py`

Implements the `Proposer` protocol. `propose(document, schema_view) -> (list[ProposedNode], list[ProposedEdge])`:

1. Get `form_spec` from `schema_view.form_spec(project=document.project)`
2. Build system prompt via `build_system_prompt(form_spec)`
3. Truncate `document.canonical_text` to ~8000 characters if needed (respect token limits)
4. Call `anthropic.Anthropic(api_key=os.environ[config.api_key_env_var]).messages.create(model=config.model, max_tokens=config.max_tokens, system=system_prompt, messages=[{"role": "user", "content": f"Extract causal claims from:\n\n{doc_text}"}])`
5. Parse JSON from response — if parsing fails, log warning and return `([], [])`
6. Build `ProposedNode` and `ProposedEdge` objects from each parsed claim
7. Catch all API exceptions (`anthropic.APIError`) — log and return `([], [])`

```
# Prompt for Claude Code:
Create apps/llm/claude_proposer.py. Add 'anthropic' to pyproject.toml dependencies.
ClaudeProposer(config: ProposerConfig) implements the Proposer protocol from
apps/llm/proposer.py. The propose(document, schema_view) method:
(1) calls schema_view.form_spec(project=document.project), (2) calls
apps.llm.prompt_builder.build_system_prompt(form_spec), (3) truncates
document.canonical_text to 8000 characters with a trailing notice if truncated,
(4) calls anthropic.Anthropic(api_key=os.environ.get(config.api_key_env_var, ''))
.messages.create(model=config.model, max_tokens=config.max_tokens,
system=system_prompt, messages=[{"role": "user", "content": doc_text}]),
(5) parses content[0].text as JSON — wrap in try/except json.JSONDecodeError and
return ([], []) on failure with a log.warning, (6) for each item in the parsed
array, create ProposedNode for subject dict and object dict (name from dict['name']
or dict.get('name', '')), ProposedEdge linking them. Return (nodes, edges). Catch
anthropic.APIError and return ([], []).
Write tests/llm/test_claude_proposer.py using unittest.mock.patch to mock
anthropic.Anthropic: test_propose_returns_correct_types (mock returns 2 claims →
2 ProposedEdge objects), test_propose_api_error_returns_empty (APIError → ([], [])),
test_propose_json_parse_error_returns_empty (non-JSON response → ([], [])),
test_propose_truncates_long_text (document with 20000-char text → API called with
≤8100 chars).
```

### Step 4 — Proposal trigger

**File:** `apps/llm/triggers.py` (new file), `apps/projects/services.py` or `apps/annotation/services.py`

`propose_for_document(document, project)`:
1. Check `ProposerConfig.objects.filter(project=project, is_enabled=True).first()`; if none, return
2. Get or create `CausalGraph` for this document
3. Instantiate `ClaudeProposer(config)`, call `.propose(document, LoomSchemaView(schema_version))`
4. For each `ProposedNode`: call `annotation.services.create_node(graph, data, origin='llm_proposed', created_by=None)` — null user is valid for machine-generated proposals
5. For each `ProposedEdge`: call `annotation.services.create_edge(graph, ...)` with `status='draft'`, `origin='llm_proposed'`
6. Cap at `config.max_proposals_per_document` edges

Hook into the assignment flow: when `trigger == 'on_assignment'`, call `propose_for_document` in a daemon thread after an annotator is assigned (find the assignment creation point in `apps/projects/services.py` or views).

```

### Step 5 — Review queue UI

**File:** `apps/llm/views.py` (new), `apps/llm/urls.py` (new), template

`ProposalReviewView` at `GET /projects/<project_pk>/review-proposals/`:
- Queries `Edge.objects.filter(graph__document__project=project, origin='llm_proposed', status='draft').select_related('subject', 'object', 'graph__document')`
- Shows each proposal as a card: subject node name, predicate (from edge.data), object node name, source_text (from `edge.data.get('original_sentence', '')`), creation timestamp
- Two HTMX buttons per card: **Accept** (POST to `/llm/proposals/<edge_pk>/accept/`) and **Reject** (POST to `/llm/proposals/<edge_pk>/reject/`)
- Accept: calls `annotation.services.advance_edge_status(edge, 'under_review', requester=request.user)`; returns HTMX swap removing the card
- Reject: calls `annotation.services.delete_edge(edge, requester=request.user)`; returns HTMX swap removing the card
- An "Edit before accepting" link opens the full annotation form pre-filled with the edge data

Access control: only assigned annotators and project admins may access.

## Acceptance Criteria

- [ ] `ClaudeProposer.propose()` works on a non-CAMO test schema (no CAMO-specific code in the prompt or parser)
- [ ] LLM is disabled by default; enabling requires explicit `ProposerConfig.is_enabled = True`
- [ ] Draft proposals never auto-advance to `complete` — annotator action required
- [ ] API key is never stored in the DB; only the env var name is stored
- [ ] Annotator can accept, reject, or edit every proposal from the review queue
- [ ] Network failure / API error returns `([], [])` without raising or blocking the UI
